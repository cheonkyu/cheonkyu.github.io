# injector vs python-dependency-injector

안녕하세요! 백엔드 개발을 하다 보면 유지보수성이 높고 테스트하기 좋은 코드를 만들기 위해 **의존성 주입(Dependency Injection, DI)**을 필연적으로 고민하게 됩니다.

Java의 Spring이나 Node.js의 NestJS 환경에서는 프레임워크가 DI를 강력하게 지원하므로 큰 고민이 없지만, Python 환경(특히 FastAPI, Flask 등)에서는 어떤 라이브러리를 도입해야 할지 망설여지곤 합니다.

많은 개발자가 NestJS나 Spring 스타일의 자동 매핑(Auto-wiring)이 지원되는 `injector`를 먼저 떠올리곤 하지만, 프로젝트가 커지고 엔터프라이즈급 아키텍처로 갈수록 **`python-dependency-injector`**가 왜 더 압도적인 선택지가 되는지 그 이유를 비교 분석해 보겠습니다.

---

## 1. 한눈에 보는 핵심 비교

| 비교 항목            | injector                                                                   | python-dependency-injector (추천)                                                  |
| :------------------- | :------------------------------------------------------------------------- | :--------------------------------------------------------------------------------- |
| **디자인 철학**      | Google Guice 스타일 기반. 타입 힌트와 생성자 데코레이터 적극 활용.         | **선언형(Declarative) 컨테이너 중심**. 소스코드와 프레임워크의 완벽한 결합 분리.   |
| **코드의 순수성**    | 클래스 내부에 `@inject` 데코레이터를 붙여야 하므로 DI 프레임워크에 종속됨. | **내부 코드는 100% 순수 파이썬(Pure Python)**으로 유지. 프레임워크 흔적이 없음.    |
| **의존성 해결 방식** | 타입 자동 매핑 (Auto-wiring). 편리하지만 거대해지면 추적이 어려움.         | **명시적 프로바이더 매핑**. 어떤 객체가 어떻게 조립되는지 완벽하게 통제 가능.      |
| **제공 기능의 깊이** | 싱글톤, 프로바이더 등 기본적인 스코프 제공.                                | **Factory, Singleton, Configuration, Resource** 등 강력한 생명주기 관리 도구 제공. |

## 2. 왜 `python-dependency-injector`가 더 강력할까?

### ① 비즈니스 로직의 완벽한 순수성 (Pure Python)

`injector` 라이브러리는 클래스 생성자에 외부 프레임워크의 데코레이터인 `@inject`를 명시해야 합니다. 이는 내 도메인/비즈니스 코드가 특정 DI 라이브러리에 종속됨을 의미합니다.
반면, `python-dependency-injector`는 내부 코드를 프레임워크의 존재조차 모르는 **순수한 상태**로 유지합니다. 덕분에 아키텍처가 프레임워크로부터 완전히 자유로워지며(Decoupled), 나중에 다른 도구로 전환하거나 CLI 스크립트, 백그라운드 워커 등으로 이식하기가 훨씬 수월합니다.

### ② "블랙 매직" 없는 명시적인 가시성과 통제력

타입 힌트 기반의 자동 매핑은 처음엔 편리하지만, 프로젝트 규모가 커지고 복잡한 인터페이스 구조를 가지게 되면 *"지금 이 타이밍에 정확히 어떤 클래스의 어떤 인스턴스가 주입되고 있는 거지?"*라는 의문이 들 때 블랙박스가 되기 쉽습니다.
`python-dependency-injector`는 `containers.py`라는 **단 하나의 지도(Map)** 안에서 모든 객체의 생성과 관계를 명시적으로 조립합니다. 디버깅이 명확해지고, 아키텍처 리뷰 시 객체 간의 의존 그래프를 한눈에 파악할 수 있습니다.

### ③ 압도적이고 정교한 생명주기 관리 (Providers)

단순한 싱글톤을 넘어 엔터프라이즈 애플리케이션에서는 다양한 객체의 생명주기 제어가 필요합니다. `python-dependency-injector`는 이를 위해 완벽한 빌딩 블록을 제공합니다.

- **`providers.Factory`**: 호출할 때마다 매번 새로운 인스턴스를 동적으로 생성합니다.
- **`providers.Singleton`**: 애플리케이션 전반에서 단 하나의 인스턴스만 공유합니다.
- **`providers.Configuration`**: 외부 `yaml`, `env` 환경 변수를 DI 컨테이너 내부에 깔끔하게 캡슐화합니다.
- **`providers.Resource`**: DB 커넥션 풀이나 외부 네트워크 소켓처럼 **초기화(Init)와 해제(Shutdown)** 처리가 엄격하게 필요한 객체의 생명주기를 완벽하게 다룹니다.

## 3. FastAPI + python-dependency-injector 실전 구현

백엔드 비즈니스 로직은 순수하게 유지하면서, FastAPI 라우터와 어떻게 깔끔하게 연동되는지 코드로 살펴보겠습니다.

### 1) app/core.py (비즈니스 로직 - 순수성 유지)

프레임워크 관련 코드가 전혀 없는, 깔끔한 순수 파이썬 클래스입니다.

```python
class UserRepository:
    def get_user_name(self, user_id: int) -> str:
        return f"User_{user_id}_from_Production_DB"

class UserService:
    # 순수 생성자 (데코레이터가 전혀 없음!)
    def __init__(self, user_repo: UserRepository):
        self.user_repo = user_repo

    def get_welcome_message(self, user_id: int) -> str:
        user_name = self.user_repo.get_user_name(user_id)
        return f"Welcome, {user_name}!"
```

### 2) app/containers.py (중앙 집중형 의존성 지도)

애플리케이션의 모든 조립 명세를 한곳에서 명시적으로 관리합니다.

```Python
from dependency_injector import containers, providers
from app.core import UserRepository, UserService

class Container(containers.DeclarativeContainer): # 의존성을 주입할 대상 파일(모듈)들을 지정
wiring_config = containers.WiringConfiguration(modules=["app.main"])

    # 의존성 정의
    user_repository = providers.Singleton(UserRepository)

    # UserService를 만들 때 어떤 인자를 주입할지 외부에서 완벽하게 통제
    user_service = providers.Factory(
        UserService,
        user_repo=user_repository
    )
```

## 3. app/main.py (FastAPI 진입점 및 라우터)

초기화 시점에 컨테이너를 빌드하고, 라우터에서는 Provide를 통해 필요한 서비스를 명시적으로 주입받습니다.

```Python
from fastapi import FastAPI
from dependency_injector.wiring import Provide, inject

from app.containers import Container
from app.core import UserService

app = FastAPI()
```

# 애플리케이션 시작 시 컨테이너 인스턴스 생성 및 바인딩

container = Container()
app.container = container # 레퍼런스 유지용

@app.get("/users/{user_id}")
@inject # 런타임 주입을 위해 라우터 함수에 데코레이터 부착
def read_user(
user_id: int,
user_service: UserService = Provide[Container.user_service]
): # 주입받은 서비스를 활용해 클린하게 비즈니스 로직 처리
return {"message": user_service.get_welcome_message(user_id)}
결론: 어떤 상황에 선택해야 할까?
injector가 어울리는 프로젝트:
소규모이거나 프로토타입 단계의 프로젝트에서, 주입 관계를 일일이 적기 귀찮고 NestJS/Spring처럼 타입 힌트 기반의 빠른 자동 매핑(Auto-wiring) 속도를 최우선으로 원할 때 편리합니다.
python-dependency-injector가 어울리는 프로젝트 (강력 추천):
대규모 서비스, 클린 아키텍처나 헥사고날 아키텍처를 지향하는 프로젝트에 적합합니다. 내 도메인 코드가 특정 라이브러리에 오염되는 것을 막고, containers.py라는 명확한 설계도 아래 전반적인 객체 그래프와 데이터베이스 풀 같은 리소스 생명주기를 완벽하게 통제하고 싶을 때 최고의 무기가 됩니다.
